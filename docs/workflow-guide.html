Projekt-Setup (einmalig)

Track A: Neues Projekt (Gruene Wiese)

  1. Brainstorming — Beschreibe deine Idee in einer Claude Code Session (mkdir projekt && cd projekt && git init reicht als Start). Brainstorming-Skill fragt Fragen, klaert Tech-Stack, schreibt Design-Doc. Am Ende weisst du welchen Stack du brauchst.
  2. /init-project <stack> — Scaffolding + Automation in einem Schritt. Erstellt das Projekt (npm init, vite create, etc.) UND richtet alles ein: Hooks, Subagents, CI, MCP, Gitignore, PR-Template, AGENTS.md, CLAUDE.md.
  3. Code schreiben — Writing Plans erstellt den Implementierungsplan, dann TDD-Zyklus. Alle Skills (TDD, Debugging, Code Review) und Hooks (Syntax, Commit-Warnung) sind ab sofort aktiv.
  4. /adapt-project — Tippe das wenn echte Source-Files und Tests existieren. Scannt Architektur und aktualisiert: Test-Coverage-Tabellen, Hook-Pfade, Subagent-Scopes, CLAUDE.md, AGENTS.md.

Track B: Bestehendes Projekt

  1. /init-project (ohne Argument) — Erkennt den vorhandenen Tech-Stack automatisch. Scannt package.json, Config-Dateien, Lock-Files. Erstellt Hooks, Subagents, CI, MCP, Gitignore, PR-Template, AGENTS.md, CLAUDE.md.
  2. /adapt-project — Sofort danach ausfuehren (Code existiert ja bereits). Scannt Architektur und aktualisiert: Test-Coverage-Tabellen, Hook-Pfade, Subagent-Scopes, CLAUDE.md, AGENTS.md.

Was laeuft im Hintergrund (Hooks)

Bei jeder Datei-Aenderung

  • JS-Dateien: node --check Syntax-Pruefung
  • .env und Lock-Files: Aenderungen blockiert

Bei jedem Commit

  • Logic-Files ohne Tests? → Warnung (domain-aware: Engine/Store vs. Frontend)
  • Frontend-Aenderungen? → E2E-Reminder
  • Keine .md-Datei im Commit? → Warnung
  • Checklist-Reminder: Tests + E2E + Docs

Browser-Testing (zwei Playwright-Integrationen)

Playwright MCP Plugin

Stellt browser_* Tools bereit (navigate, snapshot, screenshot, click, etc.). Wird von visual-testing und qa-audit fuer strukturierte Audits genutzt.

  • Snapshot-basierte Accessibility-Pruefung
  • Responsive Screenshots (resize + capture)
  • Console-Errors und Network-Failures
  • Interaktive Elemente pruefen

playwright-skill (Plugin: lackeyjb)

Claude schreibt Custom-Playwright-Scripts und fuehrt sie aus. Fuer komplexe Flows und E2E-Test-Generierung.

  • Best-Practice Locators (getByRole, getByText)
  • Proper Wait-Strategien statt Timeouts
  • Multi-Step User Flows (Login, Forms, Navigation)
  • Immer headless — kein Browser-Popup

Automatische Skills (kein manueller Aufruf noetig)

brainstorming

Triggert bei kreativer Arbeit. Fragt Fragen, schlaegt Ansaetze vor, schreibt Design-Doc.

writing-plans

Erstellt Implementierungsplan nach Design-Approval. TDD-Steps, Dateipfade, Commits.

test-driven-development

Schreibt Tests vor dem Code. Automatisch bei jeder Feature-Implementierung.

systematic-debugging

Aktiviert bei Bugs. Sucht Root Cause statt Symptome zu patchen.

requesting-code-review

Dispatcht Code-Reviewer nach jedem Task. Priorisiert Findings.

verification-before-completion

Quality Gate: Erzwingt Beweise bevor etwas als "fertig" behauptet wird.

qa-audit

Pre-PR Gate: Visual Testing + Security Review + Coverage-Gaps + Doku-Check.

finishing-a-development-branch

Branch-Abschluss: Merge lokal, PR erstellen, behalten, oder verwerfen.

playwright-skill (Plugin: lackeyjb)

Schreibt und fuehrt Playwright-Scripts aus. Best-Practice Locators (getByRole), Wait-Strategien. Immer headless. Fuer E2E-Tests und automatisierte Interaktionstests.

Subagents (bei Bedarf aufgerufen)

Test Writer

Schreibt Unit-Tests fuer neue/geaenderte Module. Kennt den Test-Style des Projekts, pflegt Coverage-Tabelle.

Security Reviewer

Nur bei Projekten mit Backend. Prueft SQL Injection, Auth, CORS, Input Validation, WebSocket Security.
